# План покращення GUI

Цей документ описує поетапний план оновлення інтерфейсу Win11 25H2 Calm Mode GUI без переписування логіки з нуля.

Головна ідея: зробити GUI зрозумілішим, спокійнішим і чеснішим щодо ризиків, зберігши сумісність із Windows PowerShell 5.1.

## Стан реалізації

- Фази 1-3 реалізовано: верхня статус-панель, двопанельна навігація, badges і risk-labels.
- Фази 4-6 реалізовано: розширена панель деталей, Apply gate після Audit, вкладки результатів `Summary` / `Details` / `Raw log`.
- Фаза 7 лишається фінальним polish етапом: масштабування, screenshots, повна візуальна перевірка й release hygiene.

## Фаза 1: Основа UI

Мета: зробити GUI схожим на нормальну програму, не змінюючи логіку твіків.

Що зробити:

- додати верхню статус-панель замість одного довгого рядка;
- показувати `Windows 11 25H2`, `Build`, `Edition`, `Mode`;
- додати badges: `Safe by default`, `Audit first`, `No changes until Apply`;
- привести шрифти, відступи, кольори до одного стилю;
- зробити нижню панель кнопок акуратнішою;
- зробити `Run Audit` головною кнопкою.

Перевірка:

- GUI запускається через `.cmd`;
- українська/англійська локалізація не ламається;
- `Audit` все ще read-only;
- Windows PowerShell 5.1 parser не показує помилок.

## Фаза 2: Нова навігація по твіках

Мета: прибрати враження "стіни checkbox-ів".

Що зробити:

- замінити або доповнити дерево двоколонковим layout:
  - зліва список категорій;
  - справа твіки вибраної категорії;
- категорії:
  - `AI / Recall`;
  - `Widgets`;
  - `Cloud Content`;
  - `Privacy`;
  - `Search`;
  - `Start / Taskbar`;
  - `Windows Update`;
  - `Edge`;
  - `Developer`;
  - `Gaming`;
  - `Optional cleanup`;
- кожен твік показувати як рядок із checkbox, назвою, коротким описом і badge;
- Appx cleanup винести в окрему категорію `Optional cleanup`.

Перевірка:

- вибір і зняття категорії працюють;
- `Select all` / `Select none` працюють;
- `Save config` / `Load config` не втрачають стан;
- фільтр працює по назві, опису і категорії.

## Фаза 3: Badges і чесна класифікація

Мета: користувач одразу бачить рівень надійності кожного твіка.

Що зробити:

- додати badge для `Confidence`:
  - `Official` - зелений;
  - `UI setting` - сірий;
  - `Requires verification` - жовтий;
  - `Best effort` - синьо-сірий;
  - `Deprecated` - помаранчевий;
- додати badge ризику:
  - `Safe`;
  - `Needs verify`;
  - `Opt-in`;
  - `May be ignored`;
- використовувати дані з `-ExportCatalog`, щоб GUI не дублював логіку руками.

Перевірка:

- кількість твіків у GUI збігається з каталогом;
- `Official`, `UISetting`, `BestEffort`, `Deprecated`, `RequiresVerification` показуються коректно;
- локалізація badge є українською й англійською.

## Фаза 4: Панель деталей твіка

Мета: дати користувачу прозорість перед застосуванням.

Що зробити:

- при кліку на твік показувати панель деталей:
  - опис;
  - registry path;
  - value name;
  - desired value;
  - confidence;
  - support / edition limitation;
  - чи потрібен admin;
  - чи потрібен reboot або explorer restart;
  - rollback note;
- для Appx cleanup показувати окреме попередження:
  - `Registry rollback does not restore removed Appx packages`.

Перевірка:

- клік по кожному твіку не спричиняє помилок;
- дані не порожні;
- довгі registry paths нормально переносяться;
- немає обрізаного тексту.

## Фаза 5: Audit flow

Мета: зробити сценарій "спочатку Audit, потім Apply" очевидним.

Що зробити:

- `Apply` вимкнений до першого успішного `Audit`;
- після `Audit` показувати summary;
- якщо є `Errors`, `Apply` лишити вимкненим або показати confirmation;
- `Undo last Apply` зробити менш акцентною кнопкою;
- `Save config` / `Load config` зробити другорядними діями.

Перевірка:

- після запуску доступний тільки безпечний шлях;
- `Apply` стає доступним тільки після Audit;
- помилка Audit не дозволяє "наосліп" застосувати зміни;
- `Undo last Apply` працює як раніше.

## Фаза 6: Result tabs

Мета: зробити результати Audit / Apply / Verify читабельними.

Що зробити:

- замість одного сирого log додати вкладки:
  - `Summary`;
  - `Details`;
  - `Raw log`;
- у `Summary` показувати лічильники:
  - `Would change`;
  - `Already OK`;
  - `Changed`;
  - `Needs verification`;
  - `Unsupported`;
  - `Errors`;
- у `Details` показувати таблицю результатів;
- `Raw log` лишити для діагностики.

Перевірка:

- Audit / Apply / Verify результати парсяться стабільно;
- raw output не втрачається;
- summary збігається з JSON / CSV report;
- великі logs не підвішують UI.

## Фаза 7: Polish і стабілізація

Мета: довести GUI до релізної якості.

Що зробити:

- перевірити масштабування 100%, 125%, 150%;
- перевірити маленьке вікно і resize;
- перевірити українські й англійські тексти;
- прибрати обрізання довгих назв;
- оновити README / docs screenshots;
- додати запис у changelog;
- запропонувати bump версії, бо це user-facing зміна.

Перевірка:

- `powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\Win11-25H2-CalmMode-GUI.ps1 -SelfTest`;
- запуск через `.cmd`;
- `Audit` без admin;
- `Audit` з admin;
- save / load config;
- language toggle;
- `git diff --check`;
- release archive містить актуальні файли.

## Рекомендований порядок реалізації

Перший крок: Фаза 1 + Фаза 3.

Причина: вони дадуть найбільший видимий ефект, але майже не зачеплять логіку вибору твіків.

Другий крок: Фаза 2 + Фаза 4.

Причина: це основна зміна UX, де треба обережно працювати зі станом checkbox-ів, фільтром і конфігами.

Третій крок: Фаза 5 + Фаза 6.

Причина: це вже покращення робочого процесу Audit / Apply / Verify та результатів.

Фінальний крок: Фаза 7.

Причина: polish краще робити після стабілізації layout і поведінки.
