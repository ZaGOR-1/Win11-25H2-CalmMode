# ROADMAP — Win11 25H2 CalmMode

План розвитку, згрупований по фазах за пріоритетом. Кожен пункт лишається в рамках філософії
проєкту: **безпека, прозорість, відкатність, opt-in для ризикованого, без зовнішніх залежностей,
сумісність із Windows PowerShell 5.1**.

Легенда ризику: 🟢 низький · 🟡 середній · 🔴 потребує обережності/дизайну.

> Статус (2026-06-19): `VERSION = 2.4` **зарелізено** (тег `v2.4` запушено). Гейти зелені —
> parse (рушій + GUI), PSScriptAnalyzer CLEAN, Pester **36/36**, Audit `EXIT=0`, forbidden patterns
> відсутні. Лишилась **Фаза 5**.

## Фаза 5 — нові фічі

Згруповано за темами. Порядок усередині — за співвідношенням цінність/ризик.

### 5A. Зручність потоку (низький ризик, висока цінність)

- [ ] **5.1 Підсумок наприкінці прогону** 🟢 *(колишнє 5.5)*
  - Друк лічильників у консоль (`WouldChange=N, Compliant=M, Skipped=K, Warning=…`); для `-WhatIf` — скільки б змінилось.
  - Синергія: GUI вже показує зведення по статусах — винести ту саму логіку в окрему функцію `Get-ResultSummary` і використати в обох місцях.

- [ ] **5.2 Виявлення pending reboot** 🟢 *(колишнє 5.3)*
  - Read-only перевірка `Component Based Servicing\RebootPending`, `WindowsUpdate\Auto Update\RebootRequired`, `PendingFileRenameOperations`.
  - Показувати у звіті окремим рядком (`Warning`/`UISetting`) — багато політик «доїжджають» лише після reboot. Нічого не перезавантажувати автоматично.

- [ ] **5.3 Покращений HTML-звіт** 🟢 *(колишнє 5.6)*
  - Секція «Needs attention» угорі (Warning / VerifyFail / RequiresVerification / MaybeIgnoredOnEdition / WouldChange) + зведення по `Confidence`.
  - Та сама модель, що вже є у GUI-таблиці, але для файлового HTML.

### 5B. Гнучкість запуску (середній ризик — нові параметри)

- [ ] **5.4 CLI-перемикачі `-Skip` / `-Only`** 🟡 *(колишнє 5.1)*
  - Швидка альтернатива `-ConfigPath` (`-Skip Widgets,Gaming` / `-Only WindowsAI`). Валідація значень через наявну `$script:BlockToggleMap`; конфлікт `-Skip`+`-Only` → помилка.
  - Тести: Pester на парсинг і на те, що `-Skip` дає 0 результатів по блоку (як уже зроблено для `-ConfigPath`).

- [ ] **5.5 Авто-`Verify` після `Apply`** 🟢 *(колишнє 5.2)*
  - Прапор `-ThenVerify` (або `-Mode All`): після Apply одразу прогнати Verify і додати його результати у звіт.
  - GUI: показувати Verify у тому ж вікні результатів одразу після Apply.

### 5C. Rollback і цілісність (потребує обережності)

- [ ] **5.6 Зручний rollback `-RestoreFrom <reportFolder>`** 🟡 *(колишнє 5.4)*
  - Імпорт згенерованого `rollback.reg` (`reg import`) без ручного пошуку файлу. У GUI — кнопка «Undo last Apply».
  - ⚠️ Чесно документувати: rollback повертає **лише registry**, не Appx-пакети. Вимагає admin. Підтвердження перед імпортом.

- [ ] **5.7 Opt-in `Enable-ComputerRestore`** 🔴
  - Прапор `-EnableSystemProtection`: якщо System Protection вимкнено, увімкнути перед створенням restore point (інакше точка тихо не створюється).
  - Системна зміна → **лише за явним прапором**, з попередженням і документацією ризику.

- [ ] **5.8 Authenticode-підпис `.ps1`** 🟡
  - Дати змогу перевірити цілісність скрипта (доповнює SHA256). Відкриває шлях до «легітимного» підписаного `.exe`-лаунчера без AV/SmartScreen-проблем.
  - Потребує сертифіката для підпису — інфраструктурне рішення, не лише код.
